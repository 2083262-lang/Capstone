1. Introduction

1.1 Purpose of This Guide

This document provides a complete, step-by-step guide for deploying the HomeEstate Realty system from a local XAMPP development environment to GoDaddy shared hosting (cPanel). It is intended for the development team and system administrators responsible for migrating the capstone project to a live production server.

The guide covers every phase of deployment, including hosting plan selection, server configuration, database migration, file upload, security hardening, performance optimization, and post-deployment verification.

1.2 Deployment Overview

The deployment process migrates the HomeEstate Realty platform from a local development server (XAMPP on Windows) to GoDaddy’s shared hosting infrastructure. The following diagram summarizes the migration path:

Summary The full deployment can be completed in approximately 1–2 hours. Steps 4.1 through 4.5 (local preparation) should be completed before beginning the GoDaddy server setup.

4. Pre-Deployment Preparation (Local)

Complete the following steps on your local machine before uploading to GoDaddy. Steps 4.1 through 4.5 can be performed in any order.

4.1 Create Database Configuration File

The current connection.php has hardcoded localhost/root credentials. Create a separate configuration file to easily switch between local and production environments.

Create config/db_config.php:

<?php
// ============================================================
// DATABASE CONFIGURATION — PRODUCTION (GoDaddy)
// IMPORTANT: Never commit this file to version control.
// ============================================================

define('DB_HOST', 'localhost');
define('DB_USERNAME', 'your_cpanel_dbuser');
define('DB_PASSWORD', 'YourStrongPassword123!');
define('DB_NAME', 'your_cpanel_prefix_realestatesystem');
GoDaddy Note GoDaddy automatically prefixes database names and usernames with your cPanel username. For example, if your cPanel username is abc123, your database will be abc123_realestatesystem and your user will be abc123_homestate.

4.2 Update Connection File

Edit connection.php:

<?php
require_once __DIR__ . '/config/db_config.php';

// Create connection using config constants
$conn = mysqli_connect(DB_HOST, DB_USERNAME, DB_PASSWORD, DB_NAME);

// Check connection
if ($conn->connect_error) {
    error_log("Database connection failed: " . $conn->connect_error);
    die("A system error occurred. Please try again later.");
}

4.3 Enable HTTPS Session Security

In login.php (line 7), uncomment the secure cookie setting to enforce HTTPS-only session cookies:

// Change this:
// ini_set('session.cookie_secure', 1);

// To this:
ini_set('session.cookie_secure', 1);

HomeEstate Realty — Deployment Guide Chapter 5: GoDaddy Server Setup

5. GoDaddy Server Setup

5.1 Set PHP Version in cPanel

Procedure: Set PHP Version
  1. Log in to your GoDaddy account → My Products → Web Hosting → Manage.
  2. Open cPanel (button at the top of the hosting dashboard).
  3. In cPanel, search for “MultiPHP Manager”.
  4. Select your domain from the list.
  5. Change PHP version to PHP 8.1 or PHP 8.2.
  6. Click Apply.
Why PHP 8.1+ The system uses random_int(), typed parameters, password_hash(), and other modern PHP features that require PHP 8.1 or later.

5.2 Configure PHP Settings (.user.ini)

GoDaddy shared hosting does not provide direct php.ini access. Instead, create a .user.ini file in your project root (inside public_html/):

; ============================================================
; PHP Settings for HomeEstate Realty (GoDaddy Production)
; ============================================================

; File uploads (system allows up to 25MB images)
upload_max_filesize = 25M
post_max_size = 30M
max_file_uploads = 20

; Execution limits
max_execution_time = 120
max_input_time = 120
memory_limit = 256M

; Error handling (NEVER show errors in production)
display_errors = Off
log_errors = On
error_reporting = E_ALL

; Session settings
session.gc_maxlifetime = 3600
session.cookie_lifetime = 0

; Security
expose_php = Off
Note After creating .user.ini, settings may take up to 5 minutes to apply on GoDaddy due to PHP configuration caching.

5.3 Create the Database

Procedure: Create MySQL Database on GoDaddy
  1. In cPanel, go to Databases → MySQL Databases.
  2. Create a New Database:
    • Database Name: type realestatesystem
    • GoDaddy will create it as: yourprefix_realestatesystem
    • Click Create Database
  3. Create a Database User:
    • Username: type homestate
    • GoDaddy will create it as: yourprefix_homestate
    • Password: click Password Generator → use a strong password → copy and save it
    • Click Create User
  4. Add User to Database:
    • Select the user (yourprefix_homestate) and database (yourprefix_realestatesystem)
    • Click Add
    • On the privileges page, check ALL PRIVILEGES
    • Click Make Changes
Write Down These 3 Values You will need them for config/db_config.php:
• Database name: yourprefix_realestatesystem
• Username: yourprefix_homestate
• Password: (the generated password)

5.4 Import the Database

Procedure: Import Database via phpMyAdmin
  1. In cPanel, go to Databases → phpMyAdmin.
  2. In the left sidebar, click on your database (yourprefix_realestatesystem).
  3. Click the Import tab.
  4. Click Choose File → select realestatesystem_export.sql from Step 4.4.
  5. Keep format as SQL.
  6. Click Go.
  7. Wait for the message: “Import has been successfully finished.”
  8. Verify: Click the database in the sidebar — you should see 30+ tables (accounts, property, agent_information, tour_requests, etc.).
If the SQL file is too large Option A: Use SSH (Deluxe plan) and run: mysql -u yourprefix_homestate -p yourprefix_realestatesystem < realestatesystem_export.sql
Option B: Split the SQL file into smaller chunks using a tool like BigDump.

6. File Upload and Configuration

6.1 Upload Project Files

Option A — cPanel File Manager (Easiest):

Procedure: Upload via cPanel File Manager
  1. In cPanel, open File Manager.
  2. Navigate to public_html/.
  3. On your local machine, select all files inside C:\xampp\htdocs\capstoneSystem\.
  4. Right-click → Send to → Compressed (zipped) folder → name it capstoneSystem.zip.
  5. In cPanel, click Upload in the toolbar.
  6. Upload the .zip file to public_html/.
  7. After upload, right-click the .zip file → Extract → extract to public_html/.
  8. Delete the .zip file after extraction.

Option B — FTP via FileZilla:

Procedure: Upload via FTP
  1. Download and install FileZilla (filezilla-project.org).
  2. Get FTP credentials from cPanel → Files → FTP Accounts.
  3. Connect using: Host = ftp.yourdomain.com, Port = 21, Username & Password = cPanel login.
  4. Navigate to public_html/ on the remote side.
  5. Drag your local project files into public_html/.
  6. Wait for all files to transfer (15–30 minutes depending on images).

6.3 Enable SSL (HTTPS)

Procedure: Enable Free SSL Certificate
  1. In cPanel, go to Security → SSL/TLS Status (or AutoSSL).
  2. Your domain should automatically have a free SSL certificate provisioned.
  3. If not, click Run AutoSSL to generate one.
  4. Wait 5–15 minutes for the certificate to be issued.
  5. Verify: Visit https://yourdomain.com — you should see the padlock icon.

6.4 Set Up Cron Job

Procedure: Configure Daily Lease Expiry Check
  1. In cPanel, go to Advanced → Cron Jobs.
  2. Under Add New Cron Job, set the schedule:
    • Minute: 0  |  Hour: 9  |  Day: *  |  Month: *  |  Weekday: *
  3. Enter the command:
/usr/local/bin/php ~/public_html/cron_lease_expiry_check.php >> ~/public_html/logs/cron.log 2>&1
GoDaddy Cron Note Use /usr/local/bin/php (not /usr/bin/php) — this is GoDaddy’s PHP binary path. The ~/ prefix refers to your home directory.

7. Security Hardening

7.1 Root .htaccess Configuration

Create a .htaccess file in the project root (inside public_html/) with the following security and performance rules:

# ============================================================
# HomeEstate Realty — Root .htaccess (Security + Performance)
# ============================================================

# ── Force HTTPS ──
RewriteEngine On
RewriteCond %{HTTPS} off
RewriteRule ^(.*)$ https://%{HTTP_HOST}%{REQUEST_URI} [L,R=301]

# ── Block access to sensitive directories ──
<IfModule mod_rewrite.c>
    RewriteRule ^config/ - [F,L]
    RewriteRule ^logs/ - [F,L]
    RewriteRule ^docs/ - [F,L]
    RewriteRule ^PHPMailer/ - [F,L]
</IfModule>

# ── Block access to hidden files ──
<FilesMatch "^\.">
    Order allow,deny
    Deny from all
</FilesMatch>

# ── Block access to sensitive file types ──
<FilesMatch "\.(sql|log|md|txt|yml|yaml|json|lock|bak|old)$">
    Order allow,deny
    Deny from all
</FilesMatch>

# ── Security Headers ──
<IfModule mod_headers.c>
    Header set X-Content-Type-Options "nosniff"
    Header set X-Frame-Options "SAMEORIGIN"
    Header set X-XSS-Protection "1; mode=block"
    Header set Referrer-Policy "strict-origin-when-cross-origin"
    Header unset X-Powered-By
    Header unset Server
</IfModule>

# ── Prevent directory listing ──
Options -Indexes

# ── Browser caching ──
<IfModule mod_expires.c>
    ExpiresActive On
    ExpiresByType image/jpeg "access plus 30 days"
    ExpiresByType image/png "access plus 30 days"
    ExpiresByType text/css "access plus 7 days"
    ExpiresByType application/javascript "access plus 7 days"
</IfModule>

# ── Gzip compression ──
<IfModule mod_deflate.c>
    AddOutputFilterByType DEFLATE text/html text/plain text/css
    AddOutputFilterByType DEFLATE application/javascript application/json
</IfModule>

10. Troubleshooting

10.1 “500 Internal Server Error”

Cause: Usually a .htaccess syntax error or PHP version mismatch.

Fix:
  1. Check logs/ for error details, or cPanel → Metrics → Errors.
  2. Temporarily rename .htaccess to .htaccess.bak to see if the error resolves.
  3. Verify PHP version is 8.1+ in cPanel → MultiPHP Manager.
  4. Check .user.ini for syntax errors.

10.2 “Database Connection Failed”

Cause: Incorrect credentials in config/db_config.php.

Fix:
  1. Verify DB_NAME includes the cPanel prefix (e.g., abc123_realestatesystem, not just realestatesystem).
  2. Verify DB_USERNAME includes the cPanel prefix (e.g., abc123_homestate).
  3. Verify the user is assigned to the database in cPanel → MySQL Databases → Current Databases.
  4. DB_HOST should be localhost (not your domain name or IP).

10.3 “2FA Emails Not Arriving”

Cause: Gmail SMTP blocked or credentials expired.

Fix:
  1. Verify Gmail app password is still valid (regenerate at myaccount.google.com → Security → App Passwords if needed).
  2. Verify two-factor authentication is enabled on the Gmail account.
  3. Try port 465 with SMTPSecure = 'ssl' as a fallback if port 587 is blocked.
  4. Alternative: Use GoDaddy’s own SMTP: smtpout.secureserver.net (port 465 SSL or 587 TLS) with a GoDaddy email address.

10.4 “File Upload Fails”

Cause: PHP upload limits not applied.

Fix:
  1. Verify .user.ini exists in the project root with correct settings.
  2. Wait 5 minutes (GoDaddy caches PHP configuration).
  3. Create a temporary phpinfo.php file and check upload_max_filesize — should show 25M.
  4. Delete phpinfo.php immediately after checking — it exposes server information.

10.5 “Images Not Displaying”

Cause: File paths or permissions incorrect.

Fix:
  1. Check that the uploads/ directory contains the image files (via cPanel → File Manager).
  2. Verify permissions are 755 on upload directories.
  3. Check the image URLs in page source — they should match the actual file paths on server.
  4. If you uploaded a .zip, ensure files extracted to the correct location (not a nested folder).

10.6 “CSS/JS Not Loading”

Cause: Asset paths incorrect.

Fix:
  1. View page source → check CSS/JS URLs are pointing to correct locations.
  2. Verify config/paths.php detects the correct BASE_URL.
  3. Check that assets/css/ and assets/js/ directories uploaded correctly.
  4. Clear browser cache (Ctrl+Shift+R).

10.7 “Cron Job Not Running”

Cause: Wrong PHP path or file path in the cron command.

Fix:
  1. In cPanel → Cron Jobs, verify the command uses /usr/local/bin/php (GoDaddy’s PHP path).
  2. Use the full absolute path: ~/public_html/cron_lease_expiry_check.php
  3. Check logs/cron.log for output.
  4. Test manually via SSH (Deluxe plan): /usr/local/bin/php ~/public_html/cron_lease_expiry_check.php
